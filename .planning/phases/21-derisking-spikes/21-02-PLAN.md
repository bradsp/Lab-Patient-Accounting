<objective>
Determine the *runtime-critical* database-object surface — the procedures, functions, and views
the live application actually calls during normal operation — versus the batch/reporting objects
that can be ported incrementally after cutover. This is the single biggest sizing lever for
Phase 23: if the runtime path touches ~5 objects, ~150 procs/UDFs become deferrable.

Purpose: §6 of the analysis names this the biggest unknown. The static catalog has ~155 procs +
~115 UDFs + ~209 views, but most are likely batch/reporting. A trace of real execution settles it.
Output: `.planning/phases/21-derisking-spikes/21-findings-runtime-surface.md` plus the reusable
trace scripts under `.planning/phases/21-derisking-spikes/spikes/`.

This plan contains a human-action checkpoint: capturing the trace requires a representative live
SQL Server environment exercised through the real Blazor + Windows-service + scheduler workflows —
there is no CLI/API substitute for "run the application against a database for a representative
window." Claude automates the scripts and the analysis; the human runs the workload.
</objective>
