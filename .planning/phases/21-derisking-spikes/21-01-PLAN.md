<task type="auto">
  <name>Task 2: MARS-dependency audit</name>
  <files>.planning/phases/21-derisking-spikes/21-findings-mars-audit.md</files>
  <action>
    `MultipleActiveResultSets=true` is set in `AppEnvironment.cs`. Npgsql has no MARS, so any code
    that iterates one query result while issuing a second query on the same connection will break.
    Search `LabBilling Library/**/*.cs` (repositories AND services) for the risky pattern: a nested
    `Query`/`Fetch`/`FetchAsync`/`Query<` call inside a `foreach`/`while`/LINQ enumeration over a
    prior `Query`/`Fetch` result, and any case where a method returns `IEnumerable`/`IQueryable`
    from a `Query<>` (deferred) that is then enumerated while another DB call runs.
    Confirm where MARS is enabled (`AppEnvironment.cs` connection-string construction, ~line 64-104)
    and grep the whole solution for other `MultipleActiveResultSets`/`MARS` occurrences.
    Note: PetaPoco's `Query<T>` streams (lazy) whereas `Fetch<T>` buffers (eager) — flag streamed
    `Query<>` results that are enumerated across nested DB calls as HIGH risk, buffered `Fetch<>`
    as LOW/none.
    Write the findings file with: (a) confirmation of where/why MARS is enabled, (b) a table of
    suspect call sites [file:line | pattern | Query vs Fetch | risk H/M/L | refactor note], (c) a
    blunt verdict — is MARS load-bearing here, or incidental? — with the count of sites that need a
    fetch-then-iterate or separate-connection refactor in Phase 22/23.
  </action>
  <verify>The findings file exists; the MARS enable site is cited with file:line; each suspect site is cited with file:line and Read-verified to actually nest a DB call inside an enumeration (no false positives in the HIGH list).</verify>
  <done>MARS enable site confirmed; suspect sites tabulated with risk levels and a stated total count of paths needing refactor; verdict on whether MARS is load-bearing is explicit.</done>
</task>
