<task type="checkpoint:decision" gate="blocking">
  <decision>Where does HL7 XML shredding live in the PostgreSQL target — in-database (`xmltable()` PL/pgSQL) or in the application tier (C# + j4jayant parser)?</decision>
  <context>This decision sets the entire scope and shape of Phase 23-02 (the largest single port, ~1,616 XML-method occurrences) and determines whether HL7 parsing logic remains split between DB and app or consolidates in the app.</context>
  <options>
    <option id="pg-xmltable">
      <name>Port to PostgreSQL xmltable()/xpath() PL/pgSQL</name>
      <pros>Smallest disturbance to current architecture (logic stays in DB); import procs keep their shape; data-tier consumers unaffected; lift-and-shift-ish per proc.</pros>
      <cons>Re-expresses 1,616 occurrences in a second XML dialect; keeps HL7 logic in the database where it is hard to unit-test; PL/pgSQL skill dependency.</cons>
    </option>
    <option id="csharp-parser">
      <name>Move shredding into C# (j4jayant parser)</name>
      <pros>Consolidates HL7 logic in the app tier where it is unit-testable and version-controlled with the rest of the code; reuses the parser already in the app; removes XML logic from the DB entirely; aligns with the Blazor/.NET direction.</pros>
      <cons>Larger up-front rewrite of the import procs into services; changes the ingestion data flow; must preserve transactional semantics the procs currently provide.</cons>
    </option>
  </options>
  <resume-signal>Select: pg-xmltable or csharp-parser (or "decide for me" to take the prototype-comparison recommendation).</resume-signal>
